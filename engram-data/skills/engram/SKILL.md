# engram 记忆系统使用规则

engram 是仿人脑的分层记忆系统：**总结当索引、按需回溯细节、遗忘=降级不删**。
本 Skill 是它的"判定大脑"——引擎（`engram` 二进制）只做机械的存取与升降级，**何时检索、写什么、判定真使用**这些判断都依本规则执行。

> 设计全貌见 `engram/docs/设计文档.md`。本文件只讲"怎么用"。

---

## 引擎二进制怎么调（codex 必读）

下文所有 `engram` 命令指**本 plugin 自带的引擎二进制**——codex 用户机器的 PATH 里通常没有 `engram`。先定位一次，之后把命令里的 `engram` 都换成它的绝对路径：
- 在本 plugin 的 `bin/` 下按平台选：Windows `bin/engram-windows-x86_64.exe`；macOS `bin/engram-macos-aarch64`（Apple 芯片）或 `bin/engram-macos-x86_64`（Intel）；Linux `bin/engram-linux-x86_64`。
- plugin 根目录：优先环境变量 `PLUGIN_ROOT`（codex 注入）；若取不到，在 `~/.codex/plugins/cache/*/engram/*/` 下找最新版本目录。
- 例：``"$PLUGIN_ROOT/bin/engram-linux-x86_64" resolve --format json``。

## 0. 作用域与库怎么定（`.engram/` 锚点）

记忆分两种库：
- **公共库**：用户级共享 L1-3，恒定加载，在 `~/.engram/general.redb`。
- **项目库**：每个项目根下的 `<项目>/.engram/engram.redb`，存该项目的 L4。

**当前是哪个项目，由「`.engram/` 锚点」决定，不靠 cwd**：引擎从当前目录**向上找最近的 `.engram/`**——
- 找到普通项目锚（`.engram/` 有库、无 `workspace` 标记）→ 那个目录即项目根；
- 找到**项目管理目录**锚（`.engram/workspace` 标记，由 `engram root` 所建）→ 项目根 = 该管理目录的**直接子目录**（你正处的那一支）；
- 一路无锚点 → 当前目录即项目根。

所以同一项目在 `src/`、`plugin/` 等任意子目录开会话，都锚定到**同一个项目库**，不再碎片化（旧的「cwd 当 root + 猜活跃子项目」已废弃）。

**怎么拿当前作用域**（手动跑 list/recall/write 前先来一次）：
```
engram resolve --format json   # 输出 general_db / project_db / project_name / kind
```
之后各命令统一带 `--general-db <general_db> --project-db <project_name>=<project_db>`。`kind` 为 `workspace` 表示你正站在项目管理目录本身（特殊，见 §1.5）。

> SessionStart hook 已按锚点自动注入热索引；只有你**手动**跑 list/recall/write 时才需先 resolve。

## 1. 会话中：怎么用记忆（主 agent）

### 读
- 热索引由 hook 在会话开始自动注入（你已能在上下文里看到 L1/L2/L3 + 各活跃项目 L4 的 cue）。**先读它**。
- **被问"这个项目是什么 / 还有哪些待做 / 进度到哪 / 以前是否处理过 X"这类问题——先查记忆、别先翻代码**：热索引里 **L4.2**（定位 / 架构地图 / 已定型决策）和 **L4.3**（未完成开口 / 待做）通常已有答案；不够就 `recall`（带 `open-loop` / `todo` 等关键词）；**仍不够才扫代码**。代码是细节 ground truth，但"还剩什么、为什么这么做、否过什么"这类**意图 / 进度问题，记忆才是第一信源**。
- 需要某条记忆的细节时：**顺着它的指针（`file:line` / doc / url）去查 ground truth**，而不是凭印象。指针就是为"验证而非重建"准备的。
- 想找没在热索引里的旧事（"我们以前是不是处理过 X"）：
  ```
  engram recall --general-db G [--project-db n=p ...] --query "关键词 关键词" [--limit 10]
  ```
  recall 默认连冷库一起搜，返回候选（id/层级/状态/score/eff/cue/指针）。**拿到候选后由你（LLM）做语义判断**挑出真正相关的，再顺指针取细节。

### 写
- **不要在干活时随手写记忆**（污染上下文、费 token）。写入集中在会话末的巩固阶段（见 §3）。
- 唯一例外：用户**显式要求**"记住 X"——这是 §2 的 override 门，可当场 `write`。

## 1.5 在「项目管理目录」里怎么办

`engram resolve` 的 `kind` 若为 `workspace`，说明你**正站在项目管理目录本身**（它只负责分门别类管理项目，不该直接堆项目记忆）。此时：

- **要做具体项目** → **主动在它下面建一个项目子目录**（按项目内容起名，拿不准就先问用户一句），进到那个子目录里干活；该项目的记忆写**它自己的库**（`resolve` 在子目录会自动锚定到它）。
- 只有**真正关于"管理这些项目"本身**的少量元信息（如这批项目的统一规范、哪个项目负责什么）才写管理目录的库——量应该很少。
- **别把某个具体项目的细节写进管理库**。

SessionStart hook 注入热索引时，也会在前言提醒你"当前在项目管理目录"。

## 2. 写什么 + 落哪层：salience 判定

分三步走：**①可恢复性闸门**（值不值得写）→ **②作用域闸门**（进公共层还是项目 L4）→ **③显著度**（importance 高低 / 落哪层）。外加一个 override（只管"写不写"）。

### ① 可恢复性闸门（写不写）
能从代码/文档/git 里 grep 出来吗？**能 → 不写**（细节留在 artifact 里，记忆只存它的补集）。不能（意图 / 为什么 / 被否的方案 / 当前进度 / 死路）→ 才有资格写。

### ② 作用域闸门（公共层 vs 项目 L4）——**默认 L4**
先问：这条**只在当前这个项目/仓库里成立**，还是**换个项目也照样成立**？
- 只在本项目成立（本仓库的架构、约定、踩过的坑、某处为何这么写）→ 写**项目 L4**（带 `--project NAME`，NAME = `engram resolve` 给的 `project_name`）。**这是默认去向。**
- 跨项目通用（用户身份/称呼/长期偏好、与具体仓库无关的工具用法或普适事实）→ 才升到**公共 L1-3**。

> 偏置很重要：**身处项目时默认 L4**，只有明确判断"拿到别的项目也有用"才进公共层。否则公共层会被单个项目的细节噪声污染，换项目后满眼无关记忆。

### ③ 显著度（importance 高低 / 落哪层）
- **【+】意外度**：推翻了默认假设 / 文档所述 / 既有认知（"该 API 实际返回 X 而非文档写的 Y"）。
- **【+】被纠正/重复次数**：用户为同一件事纠正第二次 → 重要且易忘。
- **【+】强调/命令强度**："记住""千万别""永远""重要"、祈使句。
- **【+】决策/承诺**：在若干方案里做了选择、立了承诺（artifact 只记"是什么"，不记"为什么选它、否了谁"）。

### 项目记忆该覆盖哪几类（按 L4 子层）
L4 别只记"踩坑/纠错"。每个项目应主动沉淀下面几类——都是散落 artifact 的**补集（综合 / 提炼，单条 grep 拿不到）**，是新会话快速进入状态、避免重复劳动的关键（故能过 ① 闸门）。**铁律：能从单文件 grep / `ls` 直接拿到的（目录树、配置项、CLAUDE.md 里的规则原文）不存——只存提炼出的心智模型。**

**L4.1 项目潜意识层 —— 铁律（极难忘，类比 L1，importance 0.7~0.9）**
- **项目铁律 / 不可违反的约定**：本项目必须**始终遵守**的硬规则、绝对禁忌、必守流程（如"绝不直接 push main""动这块前先读设计文档""所有 public fn 必须有 rustdoc"）。
- **怎么来**：① 用户对本项目的显式"永远 / 绝不"指令；② 被反复纠正后确立的隐性约定。**不要扫描项目去抽规则、也不要照抄 CLAUDE.md / README / lint 配置里的规则**——那些是 artifact、会被自动加载、能 grep，过不了 ① 闸门；L4.1 只存它们**没写、却踩出来**的隐性铁律。

**L4.2 项目重要层 —— 持久知识（慢衰减，importance 0.55~0.75）**
- **项目定位**：干嘛的、目标 / 范围 / 技术栈大方向 / 关键约束——提炼出"开工前必须先知道的那句话"。
- **架构 / 模块心智地图**：主要模块各干嘛、怎么协作、非显然的职责划分与"为什么这么分"。**是提炼出的心智模型，不是 `ls` 目录罗列**——哪个逻辑在哪个文件，用 `--pointer-ref` 指过去，别把目录树塞进 cue。
- **已定型 / 已辩论的决策**：哪些方案**讨论过几轮、改过、定了 X、否了 Y/Z 及原因**。**专治"后续会话又把已否决的方案重新拿出来讨论"**。

**L4.3 项目普通层 —— 临时（快衰减，importance 0.3~0.5，做完即 `supersede`）**
- **未完成的开口 / 可交接的活**：正在做、还没做完、下个会话能接着做的事；当前进度、卡在哪、下一步。蔡格尼克效应——让任意新会话无缝接力。（落 L4.3 因它衰减快：事情做完 / 废弃后自然淡出；真长期推进的开口会因反复被用而升上去。）

> **冷启动**：新项目 L4 一开始是空的。别靠引擎扫描灌库（那是"什么都往里记"的反面教材、且重复 artifact）。正确做法：首次实质会话里，主 agent 把上面 L4.1/L4.2 的定位 / 架构 / 铁律**提炼几条锚点**写下来（或用户显式"记住"），之后随使用增量补全。`claude init` 那种全项目扫描是"生成 CLAUDE.md"——是另一套机制，engram 存的是**它的补集**。

### override：用户显式"记住 X"
**只 override ①闸门 → 保证写入**；但**落哪层、给多高 importance 仍按 ②③ 判定**。"记住"= "别忘了写下来"，**不等于"这是核心身份"——不要因此直接塞进 L1，也不要把 importance 一律拉满**。

**映射到层级 / importance：**
- **L1（潜意识层，上限极小 ~7 条，极难进）**：仅限**核心身份 / 长期全局偏好**——用户是谁、怎么称呼、说什么语言、雷打不动的全局规矩。importance 0.85~1.0。**普通"记一下"的事不进 L1；拿不准就放 L2。**
- **L2（重要、跨场景复用）**：importance 0.55~0.8。
- **L3（一般、易忘）**：importance 0.0~0.4。
- **项目 L4.1/L4.2/L4.3**：所有"仅本项目"的记忆（按 ② 判定）落此，必须带 `--project NAME` 且该项目库已挂载；L4 内部再按显著度分 .1/.2/.3。

写入命令：
```
engram write --general-db G [--project-db n=p ...] \
  --level L3 --cue "一句话总结" \
  [--project NAME] [--importance 0.3] [--pinned] \
  [--pointer-kind file --pointer-ref "src/x.rs:42"] \
  [--tags intent,decision]
```
**cue 要写成"线索"**：一句话点出是什么 + 足够将来据以回想和检索的特征词；细节交给指针。

## 3. 会话末：巩固（复盘者，独立视角）

巩固由**独立的复盘者**做（不是干活的 agent 自己，避免自评注水）。复盘者**读会话转录（transcript）**回顾整场，按下面四步操作引擎。

### 3.1 判定"真使用"（决定加固谁）
真使用 = **影响了输出**的取用，分四档，只认第 3 档：
| 档 | 含义 | 算不算 |
|----|------|--------|
| 1 | 被加载进上下文 | ✗（在场 ≠ 用过） |
| 2 | 被 recall 取出但没派上用场 | ✗ |
| **3** | **真的左右了某个动作/回答** | **✓ 这才算** |
| 4 | 带来成功结果 | 不归因（太滞后） |

**铁律：与"被加载"解耦**——一条常驻 L1、每轮都被加载却没真正影响输出的记忆，也不该被加固。

从转录里挑出第 3 档的记忆 id，加固：
```
engram confirm-use --general-db G [--project-db n=p ...] --ids id1,id2,...
```
（confirm-use 会追加真使用时间戳；若该记忆在冷库，会复活成 active。）

### 3.2 纠错（superseded 硬拽）
转录里若出现"旧记忆被新信息推翻"：
```
engram supersede --general-db G [--project-db n=p ...] --id <旧> --by <新>
```
这会无视层级/floor 把旧记忆移出热层（转 superseded，留负知识痕迹）。**这是唯一能把错误从 L1 拽出来的手段。**

### 3.3 合并（一簇相关记忆冷掉时，并成粗 gist）
```
engram merge --general-db G [--project-db n=p ...] \
  --from id1,id2,id3 --cue "合并后的粗总结" [--level L2] [--importance 0.6]
```
源会转 tombstone（负知识），新记忆继承并集的访问历史。**让"实例"死、"schema"活。**

### 3.4 写入新记忆 + 升降级
- 按 §2 salience 判定，对达标的新知识 `engram write`。
- 跑一次升降级状态机（按 effective + 容量 + 迟滞自动升/降/淘汰）：
  ```
  engram consolidate --general-db G [--project-db n=p ...]   # 加 --dry-run 先看变迁
  ```

### 3.5（可选）回收
冷库/墓碑过期的硬删除（默认 cold 180 天、tombstone 3650 天；任何 recall→confirm-use 会重置计时）：
```
engram gc --general-db G [--project-db n=p ...] [--dry-run]
```

## 4. 命令速查

| 命令 | 用途 |
|------|------|
| `render` | 渲染当前热索引（hook 用它注入） |
| `recall --query` | 词法检索冷/热库返候选（不写库） |
| `write` | 新建记忆（按 §2 落层） |
| `confirm-use --ids` | 把候选确认为真使用 → 加固 / 复活 |
| `supersede --id --by` | 纠错硬拽 |
| `merge --from --cue` | 合并一簇为粗 gist |
| `consolidate` | 升降级 / 淘汰状态机（`--dry-run` 预览） |
| `gc` | TTL 硬删除（`--dry-run` 预览） |
| `list` | 可读检视（redb 是二进制，用它看内容） |
| `import --from-json-dir` | 从 JSON 目录导入（迁移/灌库用） |
| `resolve --format json\|env` | 按 `.engram/` 锚点解析当前作用域（general/project 库路径 + name + kind），手动跑读写命令前先来一次 |
| `root --project-dir D` | 把 D 设为项目管理目录（防嵌套 + 建 `.engram/workspace` 标记与管理库 + 写 L2 登记） |

> 检视任何库内容：先 `engram resolve --format json` 拿到库路径，再 `engram list --general-db G --project-db n=p --status all`
